# E2E Conversion + Deduplication Test

End-to-end test that exercises the full asset bundle conversion pipeline (real Unity, mock-aws-s3) and verifies cross-scene deduplication behavior.

## How it works

The test converts four scenes **sequentially** against the **same mock S3 store**. After each conversion it counts the files in the canonical `assets/` directory — the delta between before and after tells us how many bundles were freshly created vs reused from a previous conversion.

A final Unity EditMode test (`E2EVerification` category) loads the Cube asset bundles from Scene 1 and Scene 2 and verifies:
- Both Cubes instantiate successfully
- Both have the **same mesh** (same `Cube.gltf` source — same vertex and triangle count)
- Both have **different textures** (different `albedo.png` — pixel data comparison via `GetPixels32`, no GPU needed)

### Scene 2 — ABTestScene2.dcl.eth (worlds, 0,0)

![ABTestScene2](../docs/images/ABTestScene2.dcl.eth.png)

Second conversion. Shares most assets with Scene 1 but differs in two ways:
- `albedo.png` has a **different hash** (`bafybeich3nzq...` vs Scene 1's `bafkreigy4f55...`)
- Has `GeckoStone_01.glb` which Scene 1 doesn't

Because the **entity-wide depsDigest** is computed from ALL textures and buffers in the entity, and the texture set differs (different albedo.png + extra GeckoStone_01 deps), the digest changes. This means **every GLB/GLTF bundle gets a different composite canonical filename** and must be reconverted — even Store_02.glb and FloorBaseGrass_02.glb which don't reference albedo.png.

**What's reused (same hash, leaf assets):**
| Asset | Reason |
|---|---|
| file1.png | Same hash `bafkreiao3j5...`, bare canonical name `{hash}_{target}` |
| Floor_Grass02.png.png | Same hash `bafkreiax5kg...`, bare canonical name |

**What's fresh (new files in assets/):**
| Asset | Reason |
|---|---|
| Cube.gltf | Same hash but different depsDigest → new composite filename |
| albedo.png | Different hash entirely |
| Store_02.glb | Same hash but different depsDigest → new composite filename |
| GeckoStone_01.glb | Not in Scene 1 at all |
| FloorBaseGrass_02.glb | Same hash but different depsDigest → new composite filename |
| autogenerated-thumbnail.png | Different hash |

**Expected:** `expectedNewFiles: 12` (6 bundles × 2 variants each — raw + `.br` brotli compressed)

> **NOTE (PR #262):** With per-GLB depsDigest, Store_02.glb and FloorBaseGrass_02.glb would be reused because their per-GLB digests only include their actual dependencies (file1.png / Floor_Grass02.png.png — same in both scenes). That PR should update `expectedNewFiles` from 12 to 8 (4 bundles × 2 variants).

## Bundle naming

| Asset type | Canonical filename | Reuse behavior |
|---|---|---|
| GLB / GLTF | `{hash}_{depsDigest}_{target}` | Reused only if same hash AND same depsDigest |
| Texture / Buffer | `{hash}_{target}` | Reused if same hash (leaf — no digest) |
| .bin | Not a bundle | Inlined into the GLTF bundle by Unity |

The `depsDigest` is currently entity-wide: SHA-256 of all texture + buffer hashes in the entity. PR #262 changes this to per-GLB (only the textures/buffers that specific GLB actually references).

## CI

Runs as a single `docker run` step in `docker-common.yml`:
1. Node script: 4 conversions + file count assertions + writes bundle paths JSON
2. Unity EditMode test: loads bundles, verifies mesh/texture

The test runs first in the CI pipeline for fast feedback. During development the PR matrix is reduced to WebGL only.
